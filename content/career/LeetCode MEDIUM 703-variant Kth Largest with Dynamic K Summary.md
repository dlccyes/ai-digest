Design a data structure that supports two operations callable in any order, any number of times:

- `insert(num: int)` — add a number to the collection
- `findKthLargest(k: int)` — return the kth largest element currently in the collection, where k may differ on every call

## The Problem

The classic LeetCode 703 fixes `k` at construction time, making a min-heap of size `k` sufficient. This variant removes that constraint: `k` is a parameter of `findKthLargest`, so it can be 1 on one call and 100 on the next. The data structure must handle both operations efficiently across arbitrary sequences.

## Optimal Approach

The cleanest general solution uses a **sorted list** (order-statistics structure) backed by the `sortedcontainers.SortedList` in Python. Insert is O(log n) and finding the kth largest is O(log n) index access.

If `sortedcontainers` is unavailable, the fallback is a **plain list + heap select**: keep a raw list, and on each `findKthLargest(k)` call use `heapq.nlargest(k, self.data)[-1]`, which is O(n log k) — acceptable for moderate n.

A third option is a **balanced BST with rank augmentation** (as in an order-statistics tree), giving O(log n) for both operations, but Python has no built-in for this; `SortedList` is the practical stand-in.

**Why not a fixed min-heap?** Because k changes per query, you cannot maintain a heap of fixed size k. You would have to rebuild or resize on every call with a different k, which loses all the benefit.

## Derivation

Start from the constraint: k is unknown at insert time. That rules out any approach that preprocesses for a specific k. What you need is a structure that keeps elements in sorted order at all times. `SortedList` does exactly that: it maintains a sorted sequence under insertions in O(log n). The kth largest is then the element at index `-(k)` (i.e., `len - k` from the front), which is an O(log n) bisect-based lookup.

## Python Solution

```python
from sortedcontainers import SortedList


class KthLargestDynamic:
    """
    Data structure supporting insert and findKthLargest with variable k.

    Uses a SortedList to maintain elements in ascending order.
    - insert: O(log n)
    - findKthLargest: O(log n)
    """

    def __init__(self) -> None:
        self._data: SortedList = SortedList()

    def insert(self, num: int) -> None:
        """Add num to the collection."""
        self._data.add(num)

    def find_kth_largest(self, k: int) -> int:
        """
        Return the kth largest element (1-indexed).

        Raises IndexError if k > current size.
        """
        n = len(self._data)
        if k < 1 or k > n:
            raise IndexError(f"k={k} out of range for collection of size {n}")
        # SortedList is ascending; kth largest is at index n - k
        return self._data[n - k]


# ---------------------------------------------------------------------------
# Fallback: no external dependencies, O(n log k) per findKthLargest query
# ---------------------------------------------------------------------------
import heapq


class KthLargestDynamicFallback:
    """
    Pure-stdlib fallback using a plain list.

    - insert: O(1) amortized
    - findKthLargest: O(n log k) via heapq.nlargest
    """

    def __init__(self) -> None:
        self._data: list[int] = []

    def insert(self, num: int) -> None:
        self._data.append(num)

    def find_kth_largest(self, k: int) -> int:
        n = len(self._data)
        if k < 1 or k > n:
            raise IndexError(f"k={k} out of range for collection of size {n}")
        # nlargest returns the k largest in descending order; last element is kth
        return heapq.nlargest(k, self._data)[-1]
```

## Interview Follow-ups

### What if k is always the same across all findKthLargest calls?

If k is fixed at construction (the original LeetCode 703), a **min-heap of size k** is optimal. Maintain the invariant that the heap always holds the k largest elements seen so far. On `insert(num)`: push num; if heap size exceeds k, pop the smallest. `findKthLargest()` returns `heap[0]` in O(1). Both insert and query are O(log k), and space is O(k). This beats `SortedList` when k ≪ n.

### What is the time and space complexity of the SortedList approach?

`SortedList` from `sortedcontainers` is implemented as a list of sorted sublists (a B-tree-like structure). Insert is O(log n) amortized. Index access (`__getitem__`) is O(log n) because it must locate the correct sublist. Space is O(n). Compared to a balanced BST, the constant factors are smaller in practice because the sublists are cache-friendly arrays.

### How would you handle concurrent inserts and queries?

Wrap the structure with a `threading.Lock` (or `asyncio.Lock` for async). Acquire the lock for the full duration of both `insert` and `find_kth_largest`. For read-heavy workloads where queries vastly outnumber inserts, a `RWLock` (readers-writer lock) allows concurrent reads while serializing writes — available via third-party libraries or implementable with a `Semaphore`.

### Can you do this with O(1) findKthLargest if you accept slower inserts?

Yes, with a **skip list** or **order-statistics tree** augmented to store subtree sizes, you can achieve O(log n) for both. True O(1) `findKthLargest` for arbitrary k is not achievable with comparison-based structures — any sorted-order structure requires at least O(log n) to locate a rank. If k is always 1 (maximum), a simple running max gives O(1) query and O(1) insert, but that's a degenerate case.

### What if the stream contains duplicates?

Both `SortedList` and `heapq.nlargest` handle duplicates correctly. `SortedList.add` inserts every value including repeats. The kth largest counts by position, not by distinct value — consistent with LeetCode's convention. If the problem required the kth distinct largest, you would wrap insertions in a check against a `set` before adding to the sorted structure.

### How would you extend this to support delete(num)?

`SortedList` supports `remove(val)` in O(log n), making deletion straightforward. For the heap-based fallback, deletion is O(n) (linear scan + rebuild), which is why the heap approach is less general. With the `SortedList` variant, `delete` is a natural third operation at the same asymptotic cost as `insert`.
