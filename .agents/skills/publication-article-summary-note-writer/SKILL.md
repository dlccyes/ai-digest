# Publication Article Summary Note Writer

## Overview

Create a fresh Markdown summary note for one article from the latest issue of a recurring publication. The caller should provide the repo root, the source PDF folder, the output folder, the filename or title convention, the tag, the topic preference order, and whether the result should be committed and pushed.

## Workflow

1. Resolve the caller inputs and the issue naming rule.
2. Find the newest issue PDF in the source folder.
3. Inspect the PDF table of contents or article headings.
4. Choose the best article that is not already summarized in the output folder.
5. Create the note scaffold with the bundled script.
6. Write the summary into that file.
7. If requested, commit and push only the new note.

## Resolve Inputs

- Operate on the real repository path the caller gives you, not a temporary worktree path.
- Treat the caller instructions as authoritative for:
  - source PDF directory
  - output directory
  - filename and title format
  - tag value
  - topic preference order
  - tone or reading level
  - whether to commit and push
- If the issue identifier is not explicit, infer it from the selected PDF filename using the caller's format guidance.

## Pick The Article

- Start by finding the newest PDF (unless specified otherwise) in the source directory.
- Inspect the table of contents, article headings, or section headings before choosing.
- Search the target output directory for existing notes from the same issue and article title to avoid duplicates.
- If the best candidate is already summarized, move to the next best unsummarized article from the same latest issue.

## Create The File Scaffold

- Run the bundled script before drafting:
  `python3 /Users/dlcc/github/ai-digest/.agents/skills/publication-article-summary-note-writer/scripts/create_content_summary_stub.py --repo-root "<repo-root>" --output-dir "<output-dir>" --tag "<tag>" --generator "<tool> with <model>" "<title stem>"`
- The script creates a new file in the requested output directory and prints the created path.
- The script writes:
  - YAML frontmatter with `date`, `title`, and `tags`
  - a blank line
  - `Generated by <tool> with <model>`
  - another blank line
- Do not overwrite an existing file. The script will add a numeric suffix if a path collision happens, but you should still avoid semantic duplicates by checking issue and article title before creating the stub.

## Write The Summary

- Write directly into the created file.
- Unless the caller explicitly asks otherwise, write in third person.
- Keep the summary easy to consume, but aim at an intelligent adult reader rather than a child.
- Explain complicated ideas in plain, precise language. Translate jargon when useful, but do not lapse into baby-talk, condescension, or "explain like I'm five" phrasing.
- Make the piece stand on its own as a compact essay or blog post, not just a mechanical recap. Give it a clear structure with an engaging opening, a coherent middle, and a clean closing takeaway.
- Prefer a small number of purposeful sections over a rigid checklist of mini-headings.
- Focus on the article's main idea, the most important supporting points, and why they matter.
- Keep the prose lean. Avoid filler, hype, generic praise, and throat-clearing.
- Do not repeat the title as an H1 in the body.
- If you use headings, start at `##`.
- Leave a blank line after each heading.
- Do not use `[[wikilinks]]`.
- When writing a literal dollar amount in source Markdown, use `\\$`, not `\$`. In this Hugo setup, `\$` is reduced to `$` in rendered HTML before KaTeX runs, while `\\$` survives as `\$` and is preserved as a literal dollar sign by the site's math wrapper.


## Commit And Push

- If the caller asks to publish the result to git, stage only the created note.
- Use non-interactive git commands.
- Commit with a concise message such as `Add summary: <title>`.
- Push the current branch after committing.
- If the current branch has no upstream, use `git push -u origin HEAD`.
- Do not stage, commit, or revert unrelated changes.

## Final Checks

- Confirm the note is in the requested output directory.
- Confirm the frontmatter contains only `date`, `title`, and `tags`.
- Confirm the first content line is the generated-by line.
- Confirm the file is new and not a duplicate of an existing issue and article summary.
- Confirm any literal dollar amounts in the note body use `\\$` in the Markdown source.
- If git publish was requested, confirm the commit and push succeeded.
